# 機能要望: 強制を「バイナリ + 設定（+ git）」だけで完結させる — 生成物としての baseline を無くす

設定を育て、`baseline update` を回した段階で気づいた、esorp の**強制モデルの穴**をまとめる。個別の不満ではなく、
根はひとつ——**esorp は3層（place / form / 語彙）を持つのに、強制の表現力が足りない**。そのしわ寄せが、
`esorp.yaml` の外にもう一つ、**機械生成されて churn する状態ファイル（baseline）**を抱える形で出ている。

理想は、**バイナリと `esorp.yaml`（＋履歴は git）だけで成立する**こと。生成・再生成・ドリフトする第2の成果物を
持たないのが、ツールとして美しい。そこへ届くのに要る変更が、下の3つ。互いに噛み合った1つの設計として読める。

## 背景: なぜ今 baseline が要るのか（＝素朴に消せない理由）

baseline は「既存の違反を個体単位で猶予し、新規だけ落とす」ラチェットの土台。これは severity（ルール単位）とは
**直交**する——「このルールは enforce のまま」「でもこの既存インスタンスだけ見逃す」を同時に言えるのは baseline
だけ。だから単純に機能を消すと、**ルールを既存コードに導入/強化した瞬間に「新規は止めるが既存は猶予」が
表現できなくなる**。

普通のリンタなら代わりにコード内 `disable` コメントを使うが、**esorp はコメント衛生ツール**——抑制コメントを
撒くのは、esorp が消そうとしているノイズそのもの。だから外部ファイルに逃がすしかなかった。baseline ファイルは
esorp の原則の**帰結**であって、無駄な仕様ではない。

しかし、その「猶予」は別の形でも表現できる。それが要望3（diff モード）。3つ揃えば baseline の生成物は要らなくなる。

## 要望1: ルールごとに強度（severity）を持たせる

今は二値——違反か、baseline で黙らせるか——しかない。だから厳しすぎる構造ルール（例: `place: leading` を
一律禁止）を緩める手段が「`allow` から外して**完全に無効化**」か「baseline で**正しいコメントごと凍結**」の
両極しかなく、間が無い。実際 `leading` には**まっとうな説明コメント**が最も多く住むので、それが数千件、
hard 違反として baseline に流れ込む。

各ルール（place / form / 語彙）に `enforce / review / off` の1軸を:

```yaml
allow:
  - place: doc
    form:
      headings: enforce
      paragraphs: review     # 疑わしいが CI は落とさない
  - place: leading
    mode: review             # 禁止せず、判定は中身に委ねる
```

`esorp check` は **`enforce` の違反だけで非ゼロ終了**（RuboCop の `--fail-level`、PHPStan の level 相当）。
`review` は advisory 表示。こうすれば正当なコメントは CI を落とさず、**baseline に載る必要が消える**。

## 要望2: 焼き込みの参照検出（`form.refs`）を廃し、参照は利用者供給の語彙にする

esorp は層2について明言している——**「ツールは自前の語彙を持たない。`rules:` に書かれたものが全部」**。
`no-history` は利用者が書いた regex で、バイナリに入っていない。

ところが `form: { refs: deny }` の `refs` だけは、「参照とは `#\d+` のことだ」という**パターンをバイナリに
焼き込んでいる**。だが「何が参照か」は**プロジェクトごとに違う**（`#123` / `PROJ-456` / `GH-12` / 独自接頭辞）。
汎用ツールが特定の形を決め打ちするのは、esorp 自身の「語彙を持たない」線と食い違う。

`no-history` と同じ棚へ寄せる。置き場所非依存になり（leading に紛れた参照も捕まる）、severity で強弱も付く:

```yaml
rules:
  - id: internal-ref
    pattern: '<各プロジェクトの参照パターン>'   # ツールは知らない。利用者が書く
    severity: enforce
    message: |
      This points at a tracker a reader with only this repository cannot resolve.
```

責務は **esorp = 機構と一般原則 / プロジェクト = パターン**。ルールは `id / pattern / severity / message` の
平場でよい。「これは ref 系」というツールが解釈する分類（category / tag）は要らない——`id` が集計キー、
`message` が説明、`severity` が強さを既に持つので、その上の taxonomy は重複した間接参照になり、
「measure before adding」の原則にも反する。

## 要望3: baseline（生成ファイル）を廃し、diff / changed-lines モードで置き換える

要望1・2で baseline に載る違反はほぼ消えるが、**将来ルールを1本きつくした瞬間**にだけ「既存は猶予・新規は
落とす」が要る——ここが baseline の最後の仕事。これを**生成ファイルではなく git で**表現したい。

**base ref に対して、新規/変更したコメントだけを enforce するモード**を入れてほしい:

```
esorp check --since <base-ref>    # 変更行に現れた違反だけ enforce（既存は構造的に猶予）
```

- 既存違反は「差分に無い」ことで自動的に猶予される＝**猶予状態を記録する成果物が要らない**。
- 新規は落ちる＝ラチェットは維持。
- 普段（違反ゼロに達した後）は full-repo enforce と一致するので、モードの使い分けも意識しなくてよい。

これで **状態は git が持ち、esorp は「バイナリ + `esorp.yaml`」だけ**になる。生成・再生成・ドリフト・
マージ衝突する `*-baseline.json` を、リポジトリから恒久的に無くせる。

正直な代償: base ref への依存（マージベースをツールが知る必要がある）、既存違反の近くを触ると再浮上する churn、
そのモードでは「全違反の棚卸し」ができない（それは full-repo で別途）。＝「生成ファイル」を「git への結合」に
置き換える設計で、無条件に単純とは言い切れないが、**保守すべき第2の成果物が消える**利得は大きい。

## なぜ一般に有益か

- **severity はほぼすべての成熟したリンタが持つ軸**（RuboCop の Severity、PHPStan の level、ESLint の
  error/warn/off）。二値しか無いと、ルールが少しでも厳しいだけで baseline が「正しいコードの墓場」になる。
- **参照を語彙へ寄せる**のは、esorp が既に引いた線（語彙は利用者の物）に refs を揃えるだけ。新しい採番規約の
  たびにツールを更新せずに済む。
- **diff モード**は、baseline を「必要だが管理が重い成果物」から解放する。1・2で違反を減らし、3で最後の
  猶予手段を git に移せば、**esorp は生成物ゼロ＝バイナリと設定だけで完結する**。これは esorp が掲げる
  「設定に書かれたものが動いているものの全部・隠れた状態は無い」という思想の、素直な到達点でもある。
